# Mnemostroma — Architecture Decision Record
## Script vs Model: механика каждого слоя
## ADR-001 | v1.0 | 2026-03-09

---

## Контекст

Mnemostroma состоит из нескольких слоёв с принципиально разной механикой.
Ключевой архитектурный вопрос на каждом уровне:
**достаточно ли детерминированного скрипта, или нужна обучаемая модель?**

Критерий принятия решения:

> Если задача **детерминированная** — правило даёт правильный ответ всегда → **скрипт**.
> Если задача **паттерновая** — нужно обобщать по примерам, правило не формулируется → **модель**.

Дополнительный критерий: модель оправдана только если накоплено
достаточно данных для обучения (минимум 500+ записей).
До этого порога — модель на малых данных хуже скрипта.

---

## Принцип 1: Скрипт по умолчанию

Каждый компонент начинает как скрипт.
Переход к модели — только когда скрипт достигает потолка качества
И накоплено достаточно данных для обучения.
Это правило без исключений до v3.0.

## Решения по слоям

### ADR-001-A: Observer — скрипт ✅

**Задача:** детектировать важность, urgency, NER-сущности, теги из I/O агента.

**Решение:** детерминированный скрипт.

**Обоснование:**
- Regex + signal detection — детерминированы по определению
- GLiNER (уже в стеке) закрывает zero-shot NER без дообучения
- EmbeddingGemma (уже в стеке) закрывает векторизацию
- Никакой паттерновой задачи нет — всё формулируется правилами

**Характеристики:**
- Доп. RAM: 0MB (всё уже в стеке)
- Latency: ~0.1ms (детерм. фильтр) + ~8ms (GLiNER если нужен)
- Версия: v1.x

---

### ADR-001-B: Dissolver — скрипт ✅

**Задача:** управлять плотностью памяти (resolution 0.05–1.0) по трём осям:
время, использование, прогрессивный контекст.

**Решение:** детерминированный скрипт (математическая формула).

**Обоснование:**
- recalc() = три множителя, все считаются аналитически
- time_factor = exp(-λ × age_years) — чистая математика
- use_factor = 1 + 0.8 × log(1 + uses_per_year) — то же
- prog_factor = 1.0 - (0.3 × min(successors, 10) / 10) — то же
- Защитные флаги (milestone, principle, conflict) — булева логика

**Характеристики:**
- Доп. RAM: 0MB
- Latency: ~0.01ms
- Версия: v1.x

---

### ADR-001-C: Experience Layer — скрипт + мини-модель в v3.0

**Задача:** хранить и искать паттерны успешных и провальных решений.

**Разбивка по операциям:**

| Операция | Фаза | Решение | Обоснование |
|---|---|---|---|
| Маркировка success/failure | v1.x | Скрипт | Regex сигналы + confidence score — детерминировано |
| `content_vec` | v1.x | Скрипт | EmbeddingGemma уже есть |
| `process_vec` Фаза A | v1.x | Скрипт | embed(why_log + outcome_brief) — ретроспективно |
| Поиск по индексам | v1.x | Скрипт | HNSWlib cosine search — уже есть |
| ExperienceCluster | v1.5 | Скрипт | k-means / DBSCAN на numpy, ~5 строк |
| Maturity score | v1.5 | Скрипт | Логарифмическая формула — детерминировано |
| `process_vec` Фаза B | v1.5 | Скрипт | embed(step_log trajectory) — EmbeddingGemma |
| **`process_vec` Фаза C** | **v3.0** | **Мини-модель** | Паттерн-энкодер — обобщение поперёк доменов |

**Мини-модель Фаза C — Siamese Pattern Encoder:**
- Архитектура: Siamese transformer ~3–5M параметров
- Вход: пара (process_trajectory_A, process_trajectory_B)
- Выход: similarity score — одинаковый ли алгоритм решения
- Обучение: на накопленных парах из Experience Layer (нужно 500+)
- Экспорт: ONNX INT8
- Размер: ~5–10MB
- Latency: ~3ms
- Условие запуска: 500+ записей в Experience Layer

**Характеристики (до v3.0):**
- Доп. RAM: ~1MB (кластеры в RAM)
- Latency: ~12ms (embed) + ~1.5ms (HNSW) + ~5ms batch кластеризация
- Версия: v1.x → v1.5 → v3.0

### ADR-001-D: Subconscious Evaluator — скрипт + autoencoder в v3.0

**Задача:** рефлекторный интуитивный импульс при узнавании паттерна успеха или провала.

**Разбивка:**

| Операция | Фаза | Решение | Обоснование |
|---|---|---|---|
| DO_THIS / AVOID_THIS сигнал | v1.5 | Скрипт | cosine(new_entity, +/-exp) > threshold |
| TENSION сигнал | v1.5 | Скрипт | pos_score > T AND neg_score > T — два float |
| Адаптивный threshold по maturity | v1.5 | Скрипт | lookup table по maturity уровню |
| Cooldown / max_per_session | v1.5 | Скрипт | счётчики и timestamp |
| **Интуиция без триггера** | **v3.0** | **Мини-модель** | Аномалия в embedding-пространстве |

**Мини-модель v3.0 — Anomaly Autoencoder:**
- Архитектура: полносвязный автоэнкодер
  512d → 256 → 128 → 256 → 512 (бутылочное горло 128d)
- Обучение: только на positive experience ("норма")
- Инференс: RMSE реконструкции > threshold → аномалия → выстрел
- Семантика: "я не узнаю этот паттерн среди успешных" → сигнал осторожности
  ИЛИ: "этот паттерн очень похож на что-то успешное" → сигнал уверенности
- Экспорт: ONNX INT8
- Размер: ~2–3MB
- Latency: ~1ms
- Условие запуска: 500+ positive experience записей

**Ключевое свойство:** интуиция ошибается.
Autoencoder не даёт гарантий — он даёт вероятностный сигнал.
Это намеренно: `polarity_confidence` и `evidence_count` всегда
передаются агенту вместе с сигналом.

**Характеристики (до v3.0):**
- Доп. RAM: 0MB
- Latency: ~0.5ms (cosine + порог)
- Версия: v1.5 → v3.0

### ADR-001-E: ExperienceCluster — скрипт ✅

**Задача:** группировать записи Experience Layer в домены,
вычислять maturity (novice/competent/expert/master).

**Решение:** детерминированный скрипт.

**Алгоритм:**
```
1. Взять все process_vec из Experience Layer
2. DBSCAN(eps=0.25, min_samples=3) на numpy — автоматическое число кластеров
3. Для каждого кластера: centroid, pos/neg ratio, total_hours
4. maturity = f(log(pos_count), pos_neg_ratio) — формула
5. Пересчёт каждые 24 часа или при +10 новых записях
```

**Почему не модель для кластеризации:**
- DBSCAN на 512d векторах с numpy — достаточно точен для этой задачи
- Нет необходимости в обучении — расстояние в embedding-пространстве
  уже несёт смысловую нагрузку (EmbeddingGemma обучена на семантике)
- Модель для кластеризации потребует данных для валидации кластеров —
  которых на старте нет

**Характеристики:**
- Доп. RAM: ~1MB (центроиды + метаданные кластеров)
- Latency: ~5ms batch (пересчёт раз в 24ч)
- Версия: v1.5

## Условия перехода к моделям

Оба условия должны выполняться одновременно:

```
1. Данные:   Experience Layer >= 500 записей
             (из них >= 300 positive, >= 100 negative)

2. Качество: скрипт достиг потолка — precision@5 для process_vec
             перестала расти при добавлении новых записей

Дополнительно для Autoencoder:
3. Стабильность: кластеры не меняются более чем на 10%
                 за последние 30 дней (Experience Layer созрел)
```

До выполнения этих условий — модели не обучаются и не используются.
Скрипт продолжает работать.

## Что это означает для roadmap

```
v1.x  (сейчас → MVP)
      Всё — скрипты. Нулевые дополнительные зависимости.
      Experience Layer маркирует и ищет через существующий стек.

v1.5  (после MVP, ~200+ записей)
      ExperienceCluster (DBSCAN на numpy — уже есть).
      Базовый Subconscious Evaluator (cosine + порог).
      Всё — скрипты.

v2.0  (feedback loop)
      ctx.intuition_feedback() — калибровка порогов по реальным данным.
      Скрипт, но с адаптивными параметрами.

v3.0  (Hypómnema Strōma, 500+ записей)
      Pattern Encoder — Siamese transformer, обученный на своих данных.
      Anomaly Autoencoder — обученный на positive experience.
      Оба: ONNX INT8, ~11MB суммарно, персонализированные.
      Это не внешние модели — это интеллект выращенный из собственного опыта.
```

---

## Ключевой вывод

Mnemostroma до v3.0 — **полностью детерминированная система**.
Никакого ML обучения, никаких внешних моделей (кроме уже встроенных в стек).
Это принципиально: система работает с первой сессии, без разогрева.

В v3.0 появляются две персонализированные мини-модели (~11MB).
Они не скачиваются — они **выращиваются** из накопленного опыта пользователя.
Это и есть Hypómnema Strōma — подсознание которое нельзя скопировать.
