## Принципы приоритизации

1. **Блокеры запуска** — без них нельзя показывать публично
2. **Завершение Observer** — ядро системы, должно быть цельным
3. **Decay / Dreamer** — продуктовая ценность, но не MVP
4. **Cleanup** — параллельно с разработкой, не отдельная фаза

## ФАЗА 1 — Блокеры публичного запуска

> Без этих двух пунктов нельзя выкладывать GitHub публично.

### 1.1 Safe/Debug режим логирования
**Цель:** не создавать доверительный конфликт "offline/private" vs "пишет логи по умолчанию"

**Что сделать:**
- Добавить в `config.json`:
  ```json
  "logging": {
    "enabled": true,
    "mode": "safe",
    "db_path": "logs.db"
  }
  ```
- `safe` режим: только startup/shutdown/error события
- `debug` режим: полные 18 типов событий (текущее поведение)
- В `log_writer.py` / `log_event()`: проверять `config.logging.enabled` и `mode`
- Документировать в README что логируется и как отключить
- Тесты: проверить что `safe` режим не ломает работу системы

**Файлы:** `storage/log_writer.py`, `config.json`, `conductor.py`

### 1.2 `mnemostroma install-models` CLI
**Цель:** пользователь может установить проект в 2 команды

**Что сделать:**
- Добавить в `pyproject.toml` scripts entrypoint:
  ```toml
  [project.scripts]
  mnemostroma = "mnemostroma.__main__:main"
  ```
- Реализовать в `__main__.py` команду `install-models`:
  - Читает `models_manifest.json`
  - Скачивает через `huggingface_hub.hf_hub_download()`
  - Проверяет HF token / license acceptance
  - Показывает прогресс
  - При ошибке доступа — выводит человеческое сообщение + инструкцию
- Добавить `stack_download_manifest.md` как machine-readable конфиг для скачивания
- Тесты: mock HF download, проверка путей

**Файлы:** `pyproject.toml`, `src/mnemostroma/__main__.py`

## ФАЗА 2 — Завершение Observer (этапы B.2 / B.3)

> Anchor.flags сейчас содержат placeholder-данные. Система "притворяется" что анализирует continuation.

### 2.1 B.2 — Continuation Detector
**Статус:** архитектура полностью согласована (см. Context Transfer v4 + архив)

**Создать:** `src/mnemostroma/observer/continuation_detector.py`
```python
def detect_continuation(
    current_embedding: np.ndarray,
    current_tags: List[str],
    ctx: "SystemContext",
    max_lookback: int = 10,
    continuation_score_threshold: float = 0.65,
    related_cosine_threshold: float = 0.45,
    max_age_days: int = 7,
) -> Dict[str, Any]:
    # HNSW top-10 → age filter → combined score
    # score = cosine×0.7 + tag_overlap×0.1 + recency×0.2
    # Три состояния: continuation / related / new
    # Chain depth через anchor_index.get(sid).flags["continuation_of"]
```

**Создать:** `tests/test_continuation_detector.py`
Ключевые кейсы (11 тестов, см. Context Transfer v4):
- empty index → new entity
- high similarity → continuation (combined ≥ 0.65)
- medium similarity → related (cosine ≥ 0.45)
- low similarity → new
- self excluded
- best candidate wins
- chain depth tracing
- circular chain protection
- age filter excludes old
- tag overlap boosts score

**Изменить:** `src/mnemostroma/observer/pipeline.py`
- Шаг 6.5b: заменить placeholder-данные на вызов `detect_continuation()`
- `multi_session = text_signal OR continuation_of is not None`

**Коммит:** `feat: continuation detection — HNSW + tags + recency scoring (Stage B.2)`

## ФАЗА 4 — Decay Engine (Этап C)

> Забывание — ключевая архитектурная особенность. Без него система накапливает всё бесконечно.

### 4.1 Decay Worker
**Что сделать:**
- Фоновый asyncio worker (по аналогии с `ConsolidationWorker`)
- Интервал: configurable (default: каждые 60 минут)
- Алгоритм: экспоненциальный decay по `Dissolver` спеке
- Уровни: `decay_level` 0→1→2→3 (full→partial→skeleton→bedrock)
- Влияние `access_count` и `flags` (user_pin, outcome) на скорость decay
- Anchor на уровне 3 (bedrock) — никогда не удаляется

**Файлы:** `src/mnemostroma/subconscious/decay_engine.py`  
Интеграция: `conductor.py` (запуск worker в bootstrap)

### 4.2 Конфигурация decay
- Параметры в `config.json`: `decay_interval_min`, `decay_rate_base`, `pin_protection`
- Документировать в `config_tuner` (80 параметров)
