# Спецификация интеграции Perplexity и Model Context Protocol (MCP)

Данная спецификация составлена на основе официальной документации Perplexity по подключению кастомных коннекторов (Help Center) и официального репозитория Perplexity MCP Server на GitHub. Документ служит единым источником правды для настройки сетевого взаимодействия, протоколов авторизации и транспортного уровня при интеграции Mnemostroma и Perplexity.

---

## 1. Сценарий №1: Perplexity как MCP-клиент (Remote и Local коннекторы)

Perplexity поддерживает подключение как внешних удаленных (Remote), так и локальных (Local) MCP-серверов в качестве кастомных коннекторов (Custom Connectors) на планах Pro, Max и Enterprise. Это позволяет модели Sonar обращаться к внешним инструментам в реальном времени.

### 1.1. Параметры настройки в интерфейсе Perplexity (Settings → Connectors)

| Параметр | Обязательность | Допустимые значения / Описание |
| :--- | :--- | :--- |
| **Name** | Обязательно | Любая дружественная строка, идентифицирующая коннектор (например, `Mnemostroma Memory`). |
| **MCP Server URL** | Обязательно | Строгий публичный HTTPS-адрес удаленного MCP-сервера. **Использование незащищенного HTTP запрещено.** |
| **Description** | Опционально | Краткое описание назначения сервера и предоставляемых инструментов. |
| **Authentication** | Обязательно | Метод авторизации запросов: `None`, `API Key`, `OAuth`. |
| **Transport** | Обязательно | Протокол транспортного уровня: `Streamable HTTP` или `SSE`. |

### 1.2. Транспортный уровень: Streamable HTTP (Рекомендуемый стандарт)

Streamable HTTP является современным, стандартизованным и более надежным сетевым транспортом для удаленных MCP-серверов по сравнению с классическим SSE.

* **Единый эндпоинт (Single Endpoint):** Все входящие вызовы JSON-RPC (включая инициализацию, опрос списка инструментов `tools/list` и вызов инструментов `tools/call`) отправляются методом `POST` на один URL коннектора (например, `https://domain.com/mcp`).
* **Инфраструктурная совместимость:** Использование классических POST-запросов вместо постоянного двунаправленного SSE-соединения исключает проблемы с разрывами сессий при прохождении через обратные прокси (Nginx, Cloudflare), балансировщики нагрузки и межсетевые экраны (firewalls).
* **Автоматическое восстановление:** Поддерживает встроенные сессионные ID для быстрого бесшовного восстановления контекста при кратковременных сбоях сети.

### 1.3. Настройка через вкладку Advanced (JSON Configuration)

Помимо ручного ввода параметров в графическом интерфейсе (Name, URL, Auth), Perplexity поддерживает "продвинутую" настройку через вкладку **Advanced**. В этом режиме используется стандартный JSON-манифест (совместимый со стандартом конфигурации Claude Desktop). Этот способ является обязательным для локальных серверов и позволяет передавать дополнительные конфигурационные переменные.

Пример настройки удаленного сервера (Remote) через JSON:
```json
{
  "mcpServers": {
    "mnemostroma-remote": {
      "url": "https://mnemostroma.your-domain.com/mcp"
    }
  }
}
```

### 1.4. Локальные коннекторы (Local MCP) и Transport: `stdio`

В отличие от удаленных коннекторов, работающих по сети (Streamable HTTP / SSE), локальные коннекторы запускаются локально на устройстве в виде дочернего процесса. Обмен сообщениями JSON-RPC происходит через стандартные потоки ввода-вывода (`stdio`).

*   **Настройка:** Производится строго через вкладку **Advanced** с использованием полей `command`, `args` и `env`.
*   **Особенность npx (Подавление логов):** При вызове Node.js серверов через `npx`, **настоятельно рекомендуется** передавать флаг подавления вывода `-yq` (а не просто `-y`). Если `npx` выведет в `stdout` логи установки или предупреждения, клиент Perplexity не сможет распарсить поток как JSON-RPC и коннектор упадет с ошибкой соединения.
*   **Переменные окружения (`env`):** Блок `env` в JSON является основным способом безопасно прокинуть конфигурацию, токены и ключи доступа (например, `PERPLEXITY_API_KEY`) внутрь локального процесса MCP-сервера.

Пример конфигурации локального сервера (Local MCP):
```json
{
  "mcpServers": {
    "mnemostroma-local": {
      "command": "npx",
      "args": ["-yq", "mnemostroma-mcp-server"],
      "env": {
        "API_KEY": "your_secret_key_here",
        "PERPLEXITY_MODEL": "sonar-pro"
      }
    }
  }
}
```

## 2. Протоколы авторизации Perplexity MCP (OAuth & DCR)

При выборе типа авторизации `OAuth` Perplexity строго поддерживает два сценария интеграции в зависимости от возможностей MCP-сервера.

### 2.1. Автоматическая настройка через Dynamic Client Registration (DCR) — Режим Mnemostroma

Если MCP-сервер поддерживает стандарт DCR (RFC 7591), ручной ввод учетных данных не требуется. Поля **Client ID** и **Client Secret** в интерфейсе Perplexity **оставляются пустыми**.

#### Пошаговый флоу DCR-авторизации:
1. **Discovery (Обнаружение):** Perplexity запрашивает метаданные сервера по общеизвестному пути `GET /.well-known/oauth-authorization-server`. Сервер возвращает JSON-метаданные, содержащие адрес эндпоинта динамической регистрации в поле `"registration_endpoint"` (в Mnemostroma это `/register`).
2. **Registration (Регистрация):** Perplexity шлет POST-запрос на `registration_endpoint`, передавая информацию о себе (redirect URIs, client name). Сервер динамически генерирует уникальные `client_id` и `client_secret` и возвращает их в ответе со статусом `201 Created`.
3. **Authorization (Согласие):** Perplexity инициирует стандартный OAuth 2.0 flow: перенаправляет браузер пользователя на эндпоинт `/authorize` с полученным `client_id`. Пользователь подтверждает доступ на Consent Screen.
4. **Redirection (Редирект):** Сервер перенаправляет пользователя обратно в Perplexity с временным кодом (`code`).
5. **Token Exchange (Обмен токена):** Perplexity шлет запрос на `/token`, обменивая `code` на постоянный `access_token` для авторизации последующих запросов к MCP-инструментам.

### 2.2. Ручная настройка (Статическая регистрация)

Используется, если сервер не поддерживает DCR (например, серверы Perdoo или Elmah.io). 
* Пользователь обязан заранее сгенерировать `Client ID` и `Client Secret` на стороне MCP-сервера/провайдера авторизации.
* Полученные ключи вручную копируются и вставляются в поля конфигурации в UI Perplexity.

## 3. Сценарий №2: Perplexity как MCP-сервер (Официальный Github-проект)

Официальный репозиторий `perplexityai/modelcontextprotocol` предоставляет готовую реализацию MCP-сервера для предоставления сторонним ИИ-клиентам (Claude Desktop, Cursor и др.) доступа к поисковой платформе Perplexity.

### 3.1. Доступные инструменты (Tools)

Официальный сервер экспортирует четыре специализированных инструмента:

1. **`perplexity_search`**
   * **Назначение:** Прямой поисковый веб-запрос через Perplexity Search API.
   * **Результат:** Возвращает упорядоченный список веб-страниц с метаданными и сниппетами. Оптимален для поиска свежей информации.

2. **`perplexity_ask`**
   * **Назначение:** Быстрый диалоговый поиск и ответы на повседневные вопросы в реальном времени.
   * **Модель:** Работает на базе модели `sonar-pro`.

3. **`perplexity_research`**
   * **Назначение:** Глубокие, комплексные исследования по сложным темам с анализом множества источников.
   * **Модель:** Работает на базе модели `sonar-deep-research`. Создает развернутые аналитические отчеты.

4. **`perplexity_reason`**
   * **Назначение:** Решение сложных логических, математических и аналитических задач с пошаговыми рассуждениями.
   * **Модель:** Работает на базе модели `sonar-reasoning-pro`.

### 3.2. Управление токенами контекста: Параметр `strip_thinking`

Инструменты глубокого анализа (`perplexity_reason` и `perplexity_research`) поддерживают важнейший оптимизационный параметр конфигурации:

* **Имя параметра:** `strip_thinking` (boolean, опционально, по умолчанию `false`).
* **Влияние:** При значении `true` сервер автоматически вырезает теги рассуждений `<think>...</think>` из финального ответа перед отправкой клиенту. Это существенно снижает потребление токенов контекстного окна (context window) на стороне клиента и ускоряет обработку ответов.

## 4. Конфигурация коммуникации и безопасности

При построении или отладке сопряжения с Perplexity необходимо строго следовать следующим правилам безопасности и сетевого протокола:

1. **Обязательное использование HTTPS:** Попытки указать URL-адреса с префиксом `http://` для удаленных коннекторов блокируются в UI Perplexity.
2. **Строгая валидация JSON-RPC:** Входящие POST-запросы от Perplexity к MCP-серверу должны возвращать JSON-RPC 2.0 совместимые ответы. Любые ошибки сериализации или пустые тела ответов приводят к отключению коннектора.
3. **Механизм защиты PKCE:** Perplexity является публичным OAuth-клиентом. При интеграции с OAuth-серверами (IdP) обязательна поддержка механизма **PKCE (Proof Key for Code Exchange)** во избежание перехвата авторизационных кодов.
